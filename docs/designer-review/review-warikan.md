# デザインレビュー: 割り勘計算機 v1 (iteration 1)

## 判定: 修正要求(must 1件, should 2件, nit 1件)

## レビュー観点(今回精査した範囲)

要件整合(REQ-001〜005全件)/ タスク完遂性 / 状態網羅 / 一貫性 / アクセシビリティ / 仕様の曖昧・矛盾。全1画面・全3状態・モック3枚を精査。

## 指摘

### [must-1] 部分入力時の状態が未定義

- 該当画面・要素: 状態一覧(初期/結果/エラーの遷移)
- 問題: 「金額のみ入力・人数が空」のような部分入力時に、初期の案内文を出すのかエラーを出すのか仕様に定義がない。フロー図も「未入力→初期」「不正→エラー」のみで、空フィールドが「未入力」なのか「不正」なのか読み分けられない。実装者が質問なしで作れない
- 根拠: REQ-004-1(未入力は不正入力の一種として列挙されている)と初期状態の定義が衝突しうる / designer-ui 規約「実装者が質問なしで作れる粒度」
- 承認条件: 部分入力時(いずれかが空)の表示が状態一覧に明記されていること

### [should-1] タッチターゲットの最小サイズが未規定

- 該当画面・要素: 丸め単位・方向のセグメントボタン(モックでは高さ約33px)
- 問題: スマホ前提(REQ-005)なのに、タップ領域の最小サイズの規定がなく、モックの実寸は一般的な推奨(44px)を下回る
- 根拠: Apple HIG / WCAG 2.5.8 Target Size 相当の慣行
- 承認条件: 仕様にタッチターゲット最小44px(または相当の根拠ある値)が明記されること

### [should-2] 選択状態のセマンティクスが未規定

- 該当画面・要素: 丸め単位・方向のセグメント
- 問題: 「button でも radio でも可」とあるが、選択中であることを支援技術に伝える手段(radio の checked または aria-pressed)の要求がない。色のみの区別は不可
- 根拠: WCAG 1.4.1(色だけに依存しない)/ 4.1.2(状態のプログラム的判別)
- 承認条件: 選択状態をプログラム的に判別可能にする要求が実装への申し送りに含まれること

### [nit-1] 初期状態のプレースホルダが例値

- 該当画面・要素: 初期状態の金額フィールド placeholder「10000」
- 問題: プレースホルダの例値を実在の入力と見間違える利用者がいる(慣行上の懸念、好みの範囲)
- 根拠: なし(慣行)。承認は妨げない

## 維持すべき良い判断

- 「入力したら即答える電卓」方針(ボタンレス即時再計算)。REQ-001-2 を体験として最短で満たしており、往復中も維持すべき
- 差額表示を色+文言の二重符号化にした点(色覚依存の回避)

## 状態網羅・要件整合の確認結果

REQ-001〜005 に対応する画面要素・状態は must-1 を除き揃っている。ローディング状態の不在は根拠(同期計算)が明記されており妥当。
