# ADR-0001: ソースは3ファイル構成(index.html + app.js + warikan.js)の ESM とする

- **ステータス**: proposed
- **コンテキスト**: 実行時依存ゼロ・超小規模の静的Webアプリ。無人実装ループが「テストを実行して確認」できる構造が必要
- **決定ドライバー**: テスト可能性、内容把握のしやすさ(超小規模制約)、依存ゼロ、無人ループのタスク分割適性
- **検討した選択肢**:
  - 案A: 単一HTMLファイル(CSS/JS全埋め込み) — 利点: ファイル1つで最小 / 欠点: 計算ロジックがDOMと密結合になり単体テスト不能。ロジックとUIを別issueに分割できない
  - 案B: 3ファイル(index.html + app.js + warikan.js)、warikan.js を純粋関数の ESM に — 利点: ロジック単体テスト可、タスク分割(T1ロジック→T2UI)可能 / 欠点: ファイル数+2
  - 案C: ビルドツール(Vite等)導入 — 利点: 開発体験 / 欠点: devDependencies が発生し依存ゼロ方針に反する。この規模に過剰
- **決定**: 案B。テスト可能性とタスク分割適性が無人ループの成立条件であり、ファイル数増のコストは無視できる
- **帰結**: ロジック変更はUIに触れず可能になる / ESM のため file:// 直開きでは動かない場合がある(配信は GitHub Pages の http なので実害なし。ローカル確認は `python3 -m http.server` 等)/ ADR-0003(テスト方式)の前提になる
