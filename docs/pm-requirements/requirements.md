# 要件: 割り勘計算機(warikan)

## 背景と目的

開発パイプライン検証用の超小規模テストアプリ。飲み会等の割り勘で端数処理・幹事調整を暗算するのが面倒でミスが起きる、という課題に対し、金額と人数を入れるだけで公平な割り勘結果を即座に出す。

- 需要の根拠: **未検証**(パイプライン検証目的のため意図的にスコープ外。企画案参照)
- プロジェクト制約(全要件に適用): 完全無料 / 新規サービス登録なし / SaaS・外部API・外部CDN不使用 / 静的Web(GitHub Pages配信)/ 実行時依存ゼロ / 1画面

## 対象ユーザー

プロジェクトオーナー本人とその周辺(スマホのブラウザで使う)

## 要件一覧

### REQ-001: 割り勘計算

- ユーザーストーリー: 幹事が、会計を素早く公平に分けるために、合計金額と人数を入力して1人あたりの支払額を得られる
- 受け入れ条件:
  1. 合計金額(1〜9,999,999 の整数円)と人数(2〜99 の整数)を入力すると、1人あたりの支払額が表示される
  2. 入力値を変更すると、ページ再読み込みなしで結果が1秒以内に更新される
  3. 合計10,000円・人数4人・端数処理なし(割り切れる)の場合、1人あたり2,500円と表示される
- 依存・前提: なし

### REQ-002: 端数処理の選択

- ユーザーストーリー: 幹事が、現金で集めやすくするために、支払額の丸め単位(1円/10円/100円/500円/1000円)と丸め方向(切り上げ/切り捨て)を選べる
- 受け入れ条件:
  1. 丸め単位は 1円・10円・100円・500円・1000円 の5択、丸め方向は 切り上げ・切り捨て の2択で、初期値は「100円・切り上げ」
  2. 1人あたり支払額 = 合計÷人数 を選択単位で選択方向に丸めた金額になる(例: 合計10,000円・3人・100円・切り上げ → 3,400円)
  3. 丸め単位「1円」選択時は方向によらず床関数と天井関数が正しく適用される(例: 10,000円・3人・1円・切り捨て → 3,333円)
- 依存・前提: REQ-001

### REQ-003: 集金結果と過不足の表示

- ユーザーストーリー: 幹事が、集金後の過不足を把握するために、集金合計と合計金額との差額を確認できる
- 受け入れ条件:
  1. 集金合計(1人あたり支払額 × 人数)と、差額(集金合計 − 合計金額)が表示される
  2. 差額が正なら「◯円余る」、負なら「◯円足りない(幹事負担)」、0なら「ぴったり」と判別できる表示になる(例: 10,000円・3人・100円・切り上げ → 集金10,200円・200円余る)
- 依存・前提: REQ-001, REQ-002

### REQ-004: 入力バリデーション

- ユーザーストーリー: 利用者が、誤入力に気づけるように、不正な入力ではエラーが示され誤った結果が表示されない
- 受け入れ条件:
  1. 金額・人数が未入力・0・負数・範囲外・非整数のとき、結果の代わりに不正である旨の表示が出る
  2. 不正入力の状態で計算結果(古い結果を含む)が表示され続けない
- 依存・前提: REQ-001

### REQ-005: スマホ対応の静的配信

- ユーザーストーリー: 利用者が、その場でスマホから使えるように、公開URLをブラウザで開くだけで利用できる
- 受け入れ条件:
  1. GitHub Pages の公開URLにアクセスすると追加のインストール・登録なしで動作する
  2. 画面幅375px(スマホ相当)で横スクロールなしに全機能が操作できる
  3. 外部への通信(CDN・API・フォント含む)が一切発生しない(実行時依存ゼロ)
- 依存・前提: リポジトリ Itera-1601/my-app の GitHub Pages 設定

## スコープ外(理由つき)

- 傾斜配分(役職別の重み付け): 超小規模制約を超えるため
- 計算履歴の保存・複数会計: 保存機構(localStorage)が不要になる最小構成を優先するため
- 通貨切替・多言語: 対象ユーザーが日本円利用者のみのため
- 共有機能(URL共有・SNS): 外部連携なし制約のため
- PWA化・オフラインキャッシュ: 検証目的に不要なため
